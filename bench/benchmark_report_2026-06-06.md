# Отчет По Реализации И Тестированию Firebird Procedure Usage Counter

Дата: 2026-06-06

## 1. Цель Работ

Целью было:

- реализовать механизм подсчета использования хранимых процедур в `Firebird 3`
- проверить, как он влияет на производительность под конкурентной нагрузкой
- сравнить работу системы в двух режимах:
  - без `ProcUsageTrace`
  - с включенным `ProcUsageTrace`

## 2. Что Было Сделано

В проекте реализована гибридная схема:

- `C++` trace plugin для `Firebird 3`
- `Python`-collector для обработки spool-файлов и сохранения статистики в `SQLite`

### Реализованные компоненты

- `C++`-плагин перехватывает вызовы процедур через trace API Firebird
- вызовы агрегируются в памяти
- агрегаты сбрасываются в `JSONL` spool-файлы
- `Python`-сервис читает spool и переносит статистику в `SQLite`
- CLI позволяет:
  - инициализировать storage
  - выполнить разовый ingest
  - показать top процедур
  - показать статистику по конкретной процедуре

### Дополнительно для тестирования

Был добавлен отдельный benchmark harness:

- `bench/generate_sql.py`
- `bench/run_benchmark.py`
- `bench/README.md`

Он умеет:

- создавать отдельную benchmark-БД
- генерировать `1000` процедур
- генерировать mixed workload
- запускать параллельную нагрузку через `isql-fb`
- автоматически переключать Firebird между режимами:
  - baseline без плагина
  - режим с плагином
- перезапускать `firebird3.0`
- измерять время SQL-нагрузки
- измерять время последующего `ingest-once`
- сохранять отчет в `JSON`

## 3. Сценарий Нагрузочного Тестирования

Был использован локальный сценарий со следующими параметрами:

- `64` параллельных клиента
- `1000` процедур `BENCH_PROC_0001 .. BENCH_PROC_1000`
- `3` измерения на каждый режим
- workload смешанного типа:
  - примерно `70%` — `EXECUTE PROCEDURE`
  - примерно `20%` — `SELECT` / `UPDATE`
  - примерно `10%` — `INSERT` / `DELETE`

### Поведение процедур

Каждая процедура выполняла небольшой полезный объем работы:

- `SELECT` по ключу
- `UPDATE` счетчика и payload
- часть процедур дополнительно писала запись в логовую таблицу

### Что сравнивалось

Сравнивались две метрики:

1. время выполнения самой SQL-нагрузки
2. время полного пути для режима с плагином:
   `SQL -> spool -> ingest-once -> SQLite`

## 4. Что Было Проверено

### Автоматические тесты

Прогнаны unit-тесты Python-части:

- `python3 -m unittest discover -s tests -v`
- результат: `9/9 OK`

Покрыты проверки:

- генерация benchmark SQL-артефактов
- парсинг plugin config
- корректное переключение `firebird.conf`
- ingest spool-файлов в `SQLite`
- восстановление после `.processing`
- обработка spool в shared/sticky directory
- агрегация статистики в storage

### Интеграционные проверки

Проверено вживую:

- Firebird 3 загружает `ProcUsageTrace`
- benchmark-БД создается автоматически
- нагрузка проходит в двух режимах
- при включенном плагине создаются spool-файлы
- `ingest-once` обрабатывает spool без ошибок
- итоговая статистика попадает в `SQLite`
- число ожидаемых вызовов процедур совпадает с числом, зафиксированным в статистике

## 5. Результаты Нагрузочного Теста

Полный benchmark был выполнен с параметрами:

- `64` клиента
- `1000` процедур
- `3` итерации на режим

### Результаты без плагина

- среднее SQL-время: `5.190s`
- минимальное SQL-время: `4.977s`

### Результаты с плагином

- среднее SQL-время: `5.298s`
- минимальное SQL-время: `5.152s`
- среднее время `ingest-once`: `0.254s`

### Разница

- дополнительное среднее SQL-время: `+0.108s`
- относительный overhead плагина: `+2.08%`

## 6. Проверка Корректности Подсчета

Для benchmark-сценария было ожидаемо:

- `3584` вызова процедур на итерацию

Фактически получено:

- `3584` вызова процедур в `SQLite`

Также было подтверждено:

- все `3/3` итерации baseline валидны
- все `3/3` итерации plugin-режима валидны
- spool создавался и корректно импортировался

## 7. Итоговый Вывод

В результате работ:

- система подсчета использования процедур реализована
- она работает на реальном Firebird 3
- она выдерживает конкурентную нагрузку
- под локальным тестом overhead по SQL-времени получился небольшим: около `2%`
- этап `ingest` быстрый и не выглядит узким местом
